# MVC 架构模式(Model-View-Controller)

## 命题

> **作者主张**:MVC 是软件设计中常用的设计模式,能帮助开发人员更好地抽象和分解代码,同样也能帮助设计师将设计进行抽象和分解。

## 来源 / 出处

- 来源:第 3.1 节「利用 MVC 架构模式思考界面的设计」
- 引用:维基百科对 MVC 的标准定义

## 核心机制

```text
        ┌──────────────┐
        │  Controller  │ ←── 用户行为 / 模型变化触发
        └──────┬───────┘
               │ 控制流程、响应事件
       ┌───────┴────────┐
       ↓                ↓
┌──────────────┐  ┌──────────────┐
│     View     │←─│    Model     │
│   (视图)     │  │   (模型)     │
│  静态躯壳     │  │ 数据 + 业务   │
│ 显示模型状态   │  │ 不依赖视图     │
└──────────────┘  └──────────────┘
       ↑                │
       └────── 刷新机制 ─┘
```

### 视图(View):应用的躯壳

- **本质**:静态的「躯壳」,由各种控件和骨架元素组成,类似交互线框稿。
- **关注**:控件形状、宽高间距、颜色、动效。
- **不关注**:程序逻辑(视图一般没有程序逻辑)。
- **骨架元素**:可以固定(写死),也可以动态加载;动态加载要有缓存、Loading、404 三种本地骨架。
- **设计含义**:设计师应该用「原生控件 + 原生组件」,避免用自定义组合模拟其他平台控件,否则会丢失原生交互属性和后续维护成本。

### 模型(Model):应用的核心

- **本质**:封装业务逻辑相关的数据及对数据的处理方法。
- **关注**:界面中会变化的内容(用户信息、聊天列表、金额等)。
- **不关注**:显示方式和操作方式(模型不依赖视图和控制器)。
- **设计含义**:模型涉及的「数据」主要是「用户需要关注且会变化的内容」,固定图标文案不是模型对象。
- **性能含义**:可缓存在客户端的数据(通讯录、配置)能大幅提升加载速度,降低服务器压力。

### 控制器(Controller):交互的决策

- **本质**:起到不同层面间的组织作用,用于控制应用程序的流程。
- **关注**:事件处理(用户行为 + 数据模型变化)。
- **设计含义**:用户在界面上的每个动作(点击、滑动)都会触发控制器,控制器再决定更新哪个模型或刷新哪个视图。

## 使用步骤

1. **拿到一个应用需求**:先问「这个应用有哪些视图?」 → 列界面清单
2. **再问「这些视图需要展示什么数据?」** → 这些数据就是模型
3. **再问「用户能做什么?」** → 每个动作对应一个控制器事件
4. **再问「数据变化时,哪些视图需要刷新?」** → 建立「模型 → 视图」的刷新规则

## 适合场景

- **理解工程师的架构图**:看到一张架构图,先识别三个角色的边界。
- **设计稿评审**:能问出「这个状态改变后,哪些视图需要刷新?」这样的关键问题。
- **性能优化讨论**:能识别「这个数据能不能缓存在客户端」这样的优化机会。
- **跨平台设计**:同一套模型可以适配多个视图(iOS / Android / 桌面),减少重复设计。

## 限制与反例

- **不适用所有架构**:现代前端有 MVVM、MVP、Flux 等变体,MVC 是「入门透镜」,不是「唯一答案」。
- **过度分解**:小项目用 MVC 可能反而增加复杂度。
- **视图仍然有逻辑**:现代前端组件中视图往往带有局部状态,严格的「视图无逻辑」是理想化抽象。
