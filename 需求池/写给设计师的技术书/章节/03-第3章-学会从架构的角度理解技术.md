# 第 3 章 学会从架构的角度理解技术

## 章节定位

把计算思维落到应用层——用 MVC 架构理解应用,用 API/SDK/控件理解技术边界。

## 核心内容

### 3.1 利用 MVC 架构模式思考界面的设计

- **模型**:封装业务数据,关注变化,不依赖视图和控制器
- **视图**:静态躯壳,关注显示,一般没有程序逻辑
- **控制器**:处理事件,关注流程,起到层面间组织作用

| 角色 | 设计师关注 | 工程师关注 |
|---|---|---|
| 视图 | 控件、间距、动效、骨架元素 | 渲染、布局、状态 |
| 模型 | 哪些数据是用户关心的 | 数据库、缓存、API |
| 控制器 | 交互事件触发流程 | 事件处理、状态机 |

### 3.2 如何结合技术提升应用的用户体验

#### 3.2.1 从接口的角度构建用户体验

- 接口 = 提供能力的标准和抽象
- API(Application Programming Interface)= 应用程序编程接口
- SDK = 工具集合,封装 API + 文档 + 示例
- 对设计师的价值:减少工作量、便于与开发达成共识、证明接口不足以扩展业务

#### 3.2.2 从控件的角度设计用户体验

- 用「原生控件 + 原生组件」而非「自定义组合」
- 避免「为了炫技」放弃原生控件,后期维护成本高
- 控件缺乏原生属性 = 体验缺失 + 工作量翻倍

#### 3.2.3 从设计的角度解决性能问题

- 视图缓存:减少每次下发的资源体积
- 数据缓存:用户通讯录等不变数据放在客户端
- 骨架元素:Loading / 空白 / 404 三种本地骨架

#### 3.2.4 如何看待应用的启动和状态恢复

- 启动页 + 状态恢复 = 用户感知的「快」
- 状态恢复:让用户回到上次离开时的位置

## 关键洞察

> 设计师不应该只懂得设计视图,而是要结合技术全局看待整个应用界面该怎么设计。
