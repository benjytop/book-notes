# 跨设备动态布局

## 命题

> **作者主张**:动态布局是能跨手机、折叠屏、平板电脑、桌面计算机、车载中控系统运行的优秀应用的设计重点。需要考虑 8 种屏幕状态。

## 来源 / 出处

- 章节:第 4 章「如何为跨设备交互进行设计」,特别是 4.1 节「设计界面布局时的注意事项」

## 核心机制

### 8 种屏幕状态(必须全部覆盖)

```text
1. 手机 / 折叠屏折叠态        ← 主流移动端,需要考虑横竖屏
2. Pad / 折叠屏展开态        ← 类似小型平板,有横竖向区别
3. 折叠屏支架态              ← 半折 90°,类似笔记本
4. 多窗口态                  ← 两个窗口 1:1 / 1:3 / 2:3 等比例并排
5. 小窗态                    ← 应用在手机内以小窗口显示
6. 桌面态                    ← 应用在桌面操作系统中运行
7. Widget(小组件)           ← 主屏小部件
8. 通知                      ← 系统通知中心
```

### 三层设计决策

1. **设备形态**:手机 / 折叠 / 平板 / 桌面 / 车机,屏幕尺寸与比例不同
2. **窗口状态**:全屏 / 多窗口 / 小窗 / 支架态
3. **输入方式**:触摸 / 鼠标键盘 / 语音 / 旋钮 / 手势

## 使用步骤

1. **画 8 种状态草图**:每个核心页面都要在 8 种状态下都有可用布局
2. **识别「窄宽度」约束**:多窗口 / 小窗态下宽度被压缩,确保基本操作仍可用
3. **避免复杂操作在窄宽度下**:复杂操作留给大屏全屏态
4. **横竖屏分开设计**:不要寄希望于「自适应」,竖屏 + 横屏是两种设计
5. **桌面态必须考虑鼠标键盘**:拖动是桌面端常用操作,移动端不能用拖动作为唯一操作
6. **车机态参考 Pad**:Android 车机类似 Pad 布局,但要注意驾驶安全(对比 CarPlay 的模板化适配)

## 适合场景

- **多端产品设计**:一份代码、多端运行的应用(Flutter / React Native / Compose Multiplatform)
- **折叠屏适配**:从单形态应用升级到折叠屏应用
- **鸿蒙 / 全场景应用**:HarmonyOS 等强调「一次开发、多端部署」的生态
- **车机设计**:从移动端延伸到车机的应用

## 限制与反例

- **8 种状态不一定全部需要**:有些应用(游戏、相机)只需要全屏态,不必强求适配所有状态。
- **动态布局 ≠ 一套设计稿**:不同状态下的信息层级可能完全不同,不能用「同一个界面缩放」敷衍。
- **桌面态不是「手机加键盘」**:用户的需求和操作都不同,数据共享 ≠ 界面一致。
- **车机不是 Pad**:驾驶场景下注意力受限,不能用 Pad 思路直接套。
