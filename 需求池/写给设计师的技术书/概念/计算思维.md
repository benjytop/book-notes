# 计算思维(Computational Thinking)

## 命题

> **作者主张**:计算思维是一种解释、拆解和重构设计的思维方式,它能帮助设计师认识哪些技术可用、产品是怎么运转起来的,并指导设计师设计一系列解决方案。

## 来源 / 出处

- 作者:薛志荣(引用 7 条学术定义)
- 章节:第 2.1 节「计算思维是什么」
- 原文页内:第 2 章

## 核心机制

计算思维公认的 4 个要素,按解决复杂问题的流程配合:

```text
抽象(Abstraction) ─┐
                   ├──→ 1. 用抽象 + 分解把复杂问题拆成可管理的小块
分解(Decomposition)─┘
        ↓
模式识别(Pattern Recognition)──→ 2. 用抽象 + 模式识别理解、描述、表述问题
        ↓
算法(Algorithm) ────────────→ 3. 用模式识别理解数据相似性
                                   → 4. 用算法思维逐步解决问题
```

### 1. 抽象

- **定义**:从众多事物中抽取出共同、本质性的特征,舍弃非本质特征的过程。
- **设计师对应**:界面设计中的「视觉语言」「原子设计」「设计系统」都是抽象——把不同元素按规则分类命名。
- **关键洞察**:抽象不只是「命名」,更重要的是「暂时忽略与目标无关的细节,聚焦相关内容」。
- **典型例子**:Brad Frost 的原子设计 = 原子 / 分子 / 有机体 / 模板 / 页面。

### 2. 分解

- **定义**:把庞大或复杂的问题分而治之。
- **终止条件**:当一个问题「能通过编程和算法来解决」才算完成。这里的「编程和算法」不一定是真正的程序,可以泛指「可执行的步骤」。
- **设计师对应**:用户流程图、用户旅程地图都是分解。

### 3. 模式识别

- **定义**:在不同数据 / 场景中找到相似模式。
- **设计师对应**:设计规范 = 把成功的设计模式抽象出来复用;设计原则 = 模式背后的判断依据。

### 4. 算法

- **定义**:逐步解决问题的步骤。
- **设计师对应**:交互流程 = 算法;Design Sprint = 一套可复用的算法。

## 使用步骤

1. **面对问题时**:先用「抽象」问自己「我关心哪些维度,可以忽略哪些维度」
2. **把问题分解**:用「分解」问「能不能切成几个独立的子问题」
3. **找共性**:用「模式识别」问「这些子问题有没有相同结构」
4. **给出步骤**:用「算法」问「按什么顺序解决每个子问题」

## 适合场景

- **设计系统建设**:抽象 = 命名设计令牌(颜色 / 间距 / 字号),分解 = 把组件拆成原子 / 分子,模式识别 = 找到跨页面复用模式,算法 = 组件组合规则。
- **复杂产品设计**:用分解拆成多个模块,每个模块独立设计。
- **跨团队沟通**:用抽象后的术语(「输入」「输出」「状态」)和工程师对话。

## 限制与反例

- **抽象 ≠ 简化**:过度抽象会丢失关键细节。抽象要保留「上下文相关」的细节。
- **算法 ≠ 死板**:算法是「步骤」,不是「唯一路径」,遇到例外时分支处理。
- **计算思维 ≠ 编程**:作者在前言明确说「懂计算思维不等于懂代码」,但二者有相关性——掌握计算思维能更快学会新编程语言。
