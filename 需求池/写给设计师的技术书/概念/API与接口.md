# API 与接口

## 命题

> **作者主张**:设计师不需要懂底层代码,但需要懂 API / 接口的「输入输出」——它决定了设计师能让技术做什么、不能做什么。

## 来源 / 出处

- 章节:第 3.2 节「如何结合技术提升应用的用户体验」
- 重点小节:3.2.1「从接口的角度构建用户体验」

## 核心机制

API 在设计师视角下的简化模型:

```text
┌─────────────────────────────────────┐
│  输入(我传什么)                       │
│      ↓                              │
│   [ 黑盒 ]  ←  设计师不必知道内部实现      │
│      ↓                              │
│  输出(我得到什么)                      │
└─────────────────────────────────────┘
```

### 接口的 3 个关键问题(设计师版)

1. **我传什么**:输入是什么格式?有没有必填 / 选填?
2. **它返回什么**:成功时返回什么?失败时返回什么(错误码)?
3. **它的限制**:有没有调用频率限制?有没有并发限制?有没有地区限制?

## 使用步骤

1. **找接口文档**:每个平台都有开发者文档,设计师可以看简化版。
2. **关注输入输出**:用「自然语言翻译」理解每个参数。
3. **看错误处理**:成功之外的「失败」是设计兜底的关键。
4. **追问限制**:每个接口都有限制——问工程师「最坏情况下」会怎样。

## 适合场景

- **设计稿评审**:能问出「这个数据是从哪个接口来的?」而不是「这个数据怎么显示」。
- **跨平台设计**:同一套接口可以适配多个平台的设计。
- **个性化设计**:知道哪些参数是「用户可控」的,设计个性化方案。
- **兜底设计**:知道接口会失败,才能设计降级方案。

## 限制与反例

- **接口不 ≠ 体验**:接口再完美,如果用户不知道功能存在,体验依然为零。
- **接口更新会废弃**:作者在第 3.1.1 节举例——某些官方接口会变化甚至被取代,设计师应该用「原生控件」降低对特定接口的依赖。
- **接口 ≠ 设计自由度**:能用 API 实现 ≠ 应该用 API 实现。
