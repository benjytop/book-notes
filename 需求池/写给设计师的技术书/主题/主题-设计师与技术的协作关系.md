# 主题:设计师与技术的协作关系

## 命题

> **作者主张**:设计师要懂技术,但不是要懂代码;懂技术的目的是与工程师平等对话,提升方案通过率。

## 主要论据

### 1. 懂技术的三条理由(第 1 章)

| 理由 | 来源章节 | 关键内容 |
|---|---|---|
| 行业改变和进步 | 1.1 节 | 海外懂技术的设计师多、新一代设计师要求、互联网泡沫 + AI 同时带来挑战 |
| 技术与设计的密切关系 | 1.2 节 | 技术、设计和创新三者关系;技术架构师思维对设计师有借鉴价值 |
| 设计过程中需要考虑技术 | 1.3 节 | 前期调研、用户研究、原型设计的每一步都要考虑技术 |

### 2. 「全链路设计师」(第 1.1.2 节)

作者提出「全链路设计师」概念——设计师覆盖:

```text
用户研究 → 产品设计 → 原型设计 → 用户测试 → 开发协作 → 上线监控 → 数据分析
```

### 3. 协作而非对立(前言)

作者用自身经历说明:设计方案考虑技术实现后,评审一次通过率显著提升,减少改稿和沟通成本。

## 关键案例

- 作者在百度 / 小鹏 / 华为人机交互实验室的工作经历说明:**在实验室里设计师不懂技术,基本做不出什么东西**。
- 因为「实验室里的设计创新背后都需要有技术的支撑」,而且「日常工作中我们都需要经常和技术人员交流」。

## 设计含义

- **评审会上**:能预判「这个方案工程师做不出来」,主动调整而非等待返工。
- **沟通语言**:能用「这个数据从哪个接口来」「这个模型变化触发哪个视图更新」与工程师对话。
- **避免「炫技设计」**:知道什么是技术上做不到或成本过高的,主动避开。
- **角色定位**:设计师是「用设计语言翻译技术能力」的人,不是「画界面的美工」。
