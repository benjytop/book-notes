# 主题:技术架构思维赋能设计

## 命题

> **作者主张**:计算思维 + MVC 架构思维是设计师的「第二语言」。这两副透镜让设计师能与工程师平等对话,把抽象能力 + 分解能力 + 模式识别能力 + 算法步骤落到设计实践中。

## 主要论据

### 1. 计算思维(第 2 章)四要素

```text
抽象 → 抽取共同特征,舍弃非本质
   ↓
分解 → 把大问题拆成可管理的小问题
   ↓
模式识别 → 在不同数据中找共性
   ↓
算法 → 给出可执行的步骤
```

### 2. 架构思维:MVC(第 3 章)

- **视图(View)**:静态躯壳,关注显示
- **模型(Model)**:封装数据,关注变化
- **控制器(Controller)**:处理事件,关注流程

### 3. 从「思维」到「设计」的桥梁

| 思维工具 | 设计对应物 |
|---|---|
| 抽象 | 设计系统(命名 + 分类) |
| 分解 | 信息架构 + 组件拆分 |
| 模式识别 | 设计规范 + 复用模式 |
| 算法 | 交互流程 + Design Sprint |
| 视图 | 界面层 |
| 模型 | 数据层 |
| 控制器 | 交互层 |

## 关键洞察

- **作者明确说**:「懂计算思维不等于懂代码」。计算思维是「写文章的组织和内容」,编程是「写汉字、写英文」。
- **MVC 的核心是「关注点分离」**:视图不依赖模型,模型不依赖视图——这与设计系统的「关注点分离」一致。
- **API 是设计师的「黑盒接口」**:不必知道内部实现,只需知道输入输出与限制。

## 设计含义

- **设计系统建设**:用计算思维构建 Design System
- **设计稿评审**:用 MVC 视角问「这个状态变化触发哪些视图刷新」
- **跨平台设计**:同一模型 + 多套视图 = 跨端一致
- **性能优化讨论**:用「数据缓存」「视图骨架」与工程师对话
